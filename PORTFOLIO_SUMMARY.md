## ✍️ 자기소개서 활용 스토리 3가지

### 1️⃣ 문제 해결 능력 - "하드코딩에서 데이터 드리븐으로"

**상황**: GLOOMY 프로젝트에서 글루미의 상태(우울도, 친밀도)에 따라 40개 이상의 다른 대사를 보여줘야 했습니다.

**문제**: 처음에는 모든 대사를 if-else문과 함께 코드에 하드코딩했으나, 대사를 수정할 때마다 코드를 변경하고 다시 빌드해야 했습니다. 기획 변경이 잦아지면서 유지보수가 매우 어려웠습니다.

**해결**: 
- 대사 데이터를 JSON 파일로 분리하고, 코드는 "로드하고 표시하는 로직"만 담당하도록 재설계
- `Resources.Load<TextAsset>()`로 상황에 맞는 JSON 파일을 동적으로 로드
- 결과: 코드 재빌드 없이 JSON 파일만 수정하여 콘텐츠 업데이트 가능

**배운 점**: 콘텐츠와 로직을 분리하는 "데이터 드리븐 설계"의 중요성을 체득했습니다. 이는 확장성과 유지보수성을 동시에 높이는 핵심 원칙임을 깨달았습니다.

---

### 2️⃣ 학습 능력 - "Unity 무경험에서 완성형 프로젝트까지"

**상황**: Unity와 C# 개발 경험이 전무한 상태에서 프로젝트를 시작했습니다.

**행동**:
- Unity 공식 문서와 Learn 플랫폼을 통해 기본 개념 학습
- 문제 발생 시 Unity Answers, Stack Overflow, Discord 커뮤니티 적극 활용
- 실제 게임 코드를 분석하며 디자인 패턴 학습 (Singleton, State Machine 등)
- **제네릭, 코루틴, ScriptableObject** 등 C#과 Unity의 핵심 기능을 프로젝트에 직접 적용하며 학습

**성과**:
- DateTime을 활용한 시간 기반 시스템으로 **오프라인 진행 지원** (30분마다 위생도 감소)
- Unity Coroutine으로 우울도에 따라 **동적으로 변하는 눈물 드롭 주기** 구현
- ScriptableObject로 **비개발자도 수정 가능한 아이템 데이터 구조** 설계

**배운 점**: 공식 문서와 커뮤니티를 활용한 자기주도 학습 능력, 새로운 기술을 실제 프로젝트에 빠르게 적용하는 실행력을 키웠습니다.

---

### 3️⃣ 성장 마인드 - "작동하는 코드를 넘어 좋은 코드로"

**상황**: 첫 버전 완성 후, 코드가 "작동은 하지만" 구조가 복잡하고 수정이 어렵다는 것을 느꼈습니다.

**행동**:
- 개발한 코드를 객관적으로 복기하며 문제점 분석
- 클린 코드, 디자인 패턴 관련 자료를 찾아 학습
- **향후 개선 방향**을 구체적으로 문서화:
  - MVC/MVVM 패턴 적용으로 UI-로직 분리
  - 인터페이스 기반 의존성 주입으로 결합도 낮추기
  - 오브젝트 풀링으로 메모리 최적화
  - 유닛 테스트 추가로 코드 품질 보장

**배운 점**: 단순히 작동하는 코드가 아닌, **유지보수성과 확장성을 고려하는 개발자**가 되어야 함을 깨달았습니다. 이러한 개선점을 명확히 인지하고 문서화한 것 자체가 성장의 증거라고 생각합니다.

## 🎤 면접 예상 질문 & 답변 가이드

### Q1. "이 프로젝트에서 가장 어려웠던 기술적 문제는?"

**A**: "시간 기반 상태 변화 시스템 구현이 가장 어려웠습니다. 위생도가 30분마다 1씩 감소해야 하는데, 앱을 종료했다가 며칠 후 다시 켜도 정확히 경과 시간을 반영해야 했습니다. 

`DateTime.Now`와 저장된 마지막 업데이트 시간을 비교해 `TimeSpan`으로 경과 시간을 계산하고, 이를 30분 단위로 나눠 감소량을 적용하는 방식으로 해결했습니다. 이 과정에서 시간대 문제, 기기 시간 변경 등 엣지 케이스도 고려했습니다."

---

### Q2. "싱글톤 패턴을 사용한 이유는?"

**A**: "Unity에서 씬이 전환될 때 게임 데이터가 유지되어야 했습니다. 예를 들어 정원 씬에서 메인 씬으로 돌아와도 글루미의 상태값이 그대로 남아있어야 합니다.

싱글톤 + `DontDestroyOnLoad`를 적용한 매니저 클래스들을 만들어 전역적으로 접근 가능하면서도 씬 전환 시 파괴되지 않도록 했습니다. 

다만 싱글톤의 단점(테스트 어려움, 전역 상태 관리 복잡도)도 인지하고 있으며, 향후 의존성 주입 패턴으로 개선할 계획입니다."

---

### Q3. "제네릭을 활용한 SaveSystem의 장점은?"

**A**: "게임에는 GloomyData, GardenData, PlayerData 등 다양한 타입의 데이터가 있습니다. 제네릭을 사용하지 않으면 각 타입마다 별도의 저장/로드 함수를 만들어야 합니다.

`SaveToFile<T>()`, `LoadFromFile<T>()`로 제네릭 메서드를 구현하여 한 번의 구현으로 모든 타입에 재사용할 수 있었고, 컴파일 타임에 타입 안전성도 보장받았습니다."

```csharp
// 모든 데이터 타입에 동일하게 적용 가능
GloomyData gloomy = SaveSystem.LoadFromFile<GloomyData>("gloomy");
GardenData garden = SaveSystem.LoadFromFile<GardenData>("garden");
```

---

### Q4. "향후 개선 방향 중 우선순위는?"

**A**: "MVC/MVVM 패턴 적용을 최우선으로 생각합니다. 현재는 UI 표시, 게임 로직, 데이터 처리가 한 클래스에 섞여있어 테스트와 수정이 어렵습니다.

Model(데이터), View(UI), Controller(로직)를 분리하면:
1. 단위 테스트가 쉬워집니다 (UI 없이 로직만 테스트)
2. 협업 시 역할 분담이 명확해집니다
3. UI를 바꿔도 로직은 그대로 재사용 가능합니다

이후 DI 컨테이너를 도입해 결합도를 낮추고, 유닛 테스트 커버리지를 높일 계획입니다."
